# ralph-init

このスキルは、Ralph Runner 用の `PROMPT.md`（と必要に応じて `TODO.md`）を、プロジェクトを調査しユーザと対話しながら作成するためのものです。

プロジェクトに feature ファイル（`docs/features/` など `- [ ]` を含む `.md` ファイル群）が既にある場合は、TODO.md を作成せず、それらの feature ファイルを直接 ralph の対象として ralph-config に設定する。TODO.md を新規作成するかどうかは、プロジェクト調査の結果に基づいてユーザーと対話で決定する。

## モードの判別

呼び出し時の初期プロンプトに含まれるモード指定によって、3通りの動作をします。初期プロンプトを読んで、どのモードで起動されたかを最初に判断してください。

- 「新規モード」または初期プロンプトに `PROMPT.md の "# このプロジェクト固有のルール" を更新` のような文言 → **新規モード**
- 「サイクル継続モード」 → **サイクル継続モード**
- 「引き継ぎモード」 → **引き継ぎモード**

## 共通の禁止事項（全モード共通）

- `PROMPT.md` の `# 参照ファイル` / `# 手順` / `# 制約` セクションは絶対に編集しない（Ralph Loop の動作に必須）
- ユーザの承認なしにファイルを書き換えない
- 対話を飛ばしていきなり書き出さない
- 既存の `.bak` ファイルや `.ralph-archive/` の既存ファイルは読み取りのみ、書き換えは絶対にしない
- SCRATCH.md はこのスキルでは作成しない（`ralph` ループ実行時に自動で作られる）

## 新規モード

初回のプロジェクト初期化。`PROMPT.md` は骨格のみの状態で呼ばれる。TODO.md はまだ作成されていない。

### 責務

1. `PROMPT.md` の末尾 `# このプロジェクト固有のルール` セクション内を更新する:
   - `<!-- ralph-config ... -->` 構造化コメント: 対象ディレクトリ・ファイル・除外設定
   - プロジェクト固有ルールの箇条書き
2. **タスクファイルの方針を決定する**:
   - プロジェクトに既存の feature ファイル（`- [ ]` を含む `.md` ファイル群）がある場合 → TODO.md は作成しない。ralph-config の `TODO_DIRS` にそのディレクトリを設定する
   - feature ファイルがない場合 → `TODO.md` を新規作成し、プロジェクト固有のタスクリストを記載する
   - どちらにするかは必ずユーザーに確認する

### 手順

**ステップ1: プロジェクト調査（自動、ユーザに聞かない範囲）**

- ディレクトリ構造（`ls`、主要なサブディレクトリ）
- プロジェクトタイプ: `package.json` / `pyproject.toml` / `Cargo.toml` / `go.mod` / `Gemfile` などの有無
- `README.md` の内容
- 既存の `CONTRIBUTING.md`, `CLAUDE.md`, `AGENTS.md`
- `.gitignore` から推測できる生成物
- 既存のテストディレクトリ・フレームワーク
- **TODO やタスクを含む `.md` ファイルの探索**: `docs/` 配下のサブディレクトリ、`- [ ]` を含むファイルを探す

調査結果は簡潔にユーザへ報告（長文にしない）。

**ステップ2: ユーザへの質問（1〜3問ずつ刻む）**

1. 目的: Ralph Runner を使って何を進めたいか
2. 完成の定義: 何が出来れば完了とみなせるか
3. **タスクファイルの方針**:
   - feature ファイルが見つかった場合: 「`docs/features/` にタスクを含む feature ファイルが見つかりました。これらを ralph の進捗ファイルとして直接使いますか？ それとも TODO.md を別途作成しますか？」
   - feature ファイルが見つからない場合: 「TODO.md を作成してタスクを管理します」と伝える（確認のみ）
4. **対象ファイルの確認**（feature ファイルを使う場合）: 「除外したいディレクトリはありますか？ 他に対象に含めたいファイルはありますか？」
5. コーディング規約: 言語、テストフレームワーク、Linter など（推測できた分は確認のみ）
6. 触ってはいけない領域: 特定ディレクトリやファイル
7. コミット規約: メッセージ形式など
8. その他の制約

質問しすぎない。README やコードから明らかな項目は提示して「この理解で合っていますか？」と確認するだけで済ませる。

**ステップ3: 提案**

- **ralph-config 構造化コメント草案**: ステップ2で確認した対象ディレクトリ・除外設定を以下の形式で提示:
  ```
  <!-- ralph-config
  TODO_DIRS=docs/features docs/nfr
  TODO_FILES=docs/architecture.md
  EXCLUDE_DIRS=docs/legacy docs/notes
  -->
  ```
  - `TODO_DIRS`: スキャン対象ディレクトリ（スペース区切り、再帰的にスキャン）
  - `TODO_FILES`: 個別の対象ファイル（スペース区切り）
  - `EXCLUDE_DIRS`: 除外ディレクトリ（スペース区切り）
- **PROMPT.md の追加セクション草案**: プロジェクト固有ルールの箇条書き
- **TODO.md の草案**（feature ファイルを使わない場合のみ）: `- [ ]` 形式。粒度は「1周の Ralph Loop で1つ処理できる」程度（1〜数時間で終わる単位）
- feature ファイルを使う場合は TODO.md の草案は不要。ralph-config の設定だけで十分

**ステップ4: 修正サイクル**

ユーザの修正指示を反映し再提示。納得するまで繰り返す。

**ステップ5: ファイル書き出し**

明示的な承認を得てから:

1. `PROMPT.md` の `# このプロジェクト固有のルール` 以降を、確定内容で置換（それ以外は触らない）。以下の順で書き出す:
   - `<!-- ralph-config ... -->` 構造化コメント（bash が解析するため、フォーマットを厳守: 1行1設定、`KEY=value` 形式）
   - プロジェクト固有ルールの箇条書き（自然言語）
2. TODO.md を作成する方針の場合のみ、`TODO.md` を確定内容で作成する
   - feature ファイルを使う方針の場合は TODO.md を作成しない

完了後「`ralph` を実行するとループが始まります」と伝えて終了。

## サイクル継続モード

前サイクルが完了した状態で呼ばれる。`PROMPT.md` はプロジェクト固有化済み、`TODO.md` は雛形に戻っている。`.ralph-archive/` に前サイクル分のアーカイブがある。

### 責務

- `PROMPT.md` は**絶対に触らない**
- `.ralph-archive/` 内の最新アーカイブ（`TODO-*.md` と `SCRATCH-*.md`）を読んで前サイクルの成果を把握し、次サイクルの `TODO.md` を作成する

### 手順

**ステップ1: 前サイクルの把握**

- `.ralph-archive/` を `ls` で一覧
- 最新タイムスタンプの `TODO-*.md` と `SCRATCH-*.md` を読む
- 何が完了したか、どんな判断があったか、SCRATCH.md 内の `QUESTION:` で未解決の論点を把握
- 現プロジェクトの `PROMPT.md` も読み、プロジェクト固有ルールを尊重

**ステップ2: ユーザへの確認と質問**

- 前サイクルで完了した範囲を1〜2行で要約して示す
- 「次サイクルで進めたいことは何ですか？」「前サイクルで残った課題はありますか？」と質問
- 必要に応じて過去アーカイブ（もっと古いもの）も読んで文脈を補う

**ステップ3: 提案**

- 次サイクルの `TODO.md` 草案を提示

**ステップ4: 修正サイクル**

新規モードと同様。

**ステップ5: ファイル書き出し**

承認後、`TODO.md` のみを書き換える。`PROMPT.md` は触らない。

---

## 引き継ぎモード

サイクル途中で `ralph init` が呼ばれ、未完了タスク `[ ]` を新サイクルに引き継ぐ場合のモード。

### 前提

- 現在の `TODO.md` に未完了 `[ ]` と完了 `[x]` が混在している
- 同じ内容のスナップショットが `.ralph-archive/TODO-*-inherited.md` に保存されている（読み取り用）
- `PROMPT.md` はそのまま

### 責務

- `PROMPT.md` は**絶対に触らない**
- 現 `TODO.md` から未完了 `[ ]` タスクを抽出し、整理・追加提案を対話で行い、新しい `TODO.md` を作成する

### 手順

**ステップ1: 未完了タスクの把握**

- 現 `TODO.md` を読み、`- [ ]` の行をすべて抽出
- 完了済み `- [x]` は無視（参考情報として把握はしてよい）
- SCRATCH.md があれば読み、直近の作業文脈を把握

**ステップ2: ユーザへの確認**

- 未完了タスク一覧を提示して、以下を確認:
  - そのまま引き継ぐか、粒度を見直すか
  - 並び順を変えたいか
  - 追加したいタスクはあるか
  - 不要になって削除したいタスクはあるか

**ステップ3: 提案**

- 整理後の `TODO.md` 草案を提示

**ステップ4: 修正サイクル**

新規モードと同様。

**ステップ5: ファイル書き出し**

承認後、`TODO.md` のみを書き換える。
